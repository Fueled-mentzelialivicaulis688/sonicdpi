# DPI scene snapshot — May 2026

This file tracks what the Russian TSPU + RKN-mandated DPI is doing
**right now** vs. what SonicDPI's engine can express, and the
profile changes that were made in this round of catch-up. Update
whenever Flowseal/Dronatar publishes a new `general (Dronatar) vN.x`
or zapret upstream lands a new `--dpi-desync` mode.

## Reference profile we're tracking

[`general (Dronatar) v4.6` — 13.02.2026](https://github.com/Flowseal/zapret-discord-youtube/discussions/3279).
It supersedes v4.4/v4.5 but keeps most of their structure. The key
shifts since v3.x (which is what SonicDPI's older defaults mirrored):

| Lane                          | v3.x (old SonicDPI default) | v4.6 (current effective)                                                           |
|-------------------------------|------------------------------|------------------------------------------------------------------------------------|
| YouTube TCP/443               | `multisplit + md5sig`        | `multidisorder` + `split-pos=1,midsld` + `seqovl=681` + google-CH pattern          |
| Discord TCP/443               | `multisplit`                 | `fake` + `fake-tls-mod=rnd,dupsid,sni=vk.me` + `fooling=badseq` + bs-inc=0/ba-inc=1 |
| Discord port set              | 443 only                     | 443, 2053, 2083, 2087, 2096, 8443                                                   |
| YouTube QUIC (UDP/443)        | `fake repeats=6`             | `fake repeats=11` + google-QUIC Initial                                             |
| Discord voice (UDP)           | dbankcloud-QUIC fakes        | google-QUIC Initial as both `fake-stun` and `fake-discord`                          |
| HTTP/80 (googlevideo)         | (none)                       | `fake` + `fake-http=tls_clienthello_www_google_com.bin` + `fooling=badseq`         |
| `list-general+Extras` (TLS)   | (none)                       | `fake,multisplit` + `sni=vk.me` + `split-pos=1,host` + bs-inc=0/ba-inc=1            |
| Cloudflare WARP gateway       | (none)                       | `syndata` + `fake-syndata=0x00` + `cutoff=n2`                                       |
| TLS-fake SNI rewrite target   | `www.google.com`             | `vk.me` (RU-ASN, currently TSPU-whitelisted)                                       |

## What landed in this update

In `crates/sonicdpi-engine/src/profile.rs`, built-in `youtube-discord`:

- QUIC repeats: `6 → 11`. Below 8 the TSPU pre-Initial drop swallows
  the burst before the real Initial reaches the server.
- `discord_voice_prime` default fake payloads: `dbankcloud-QUIC + stun.bin`
  → `quic_initial_www_google_com.bin` for **both** `fake_stun` and
  `fake_discord`. The captured RU-bank QUIC stopped passing TSPU
  around autumn 2025.

### 2026-05-08: Discord-Cloudflare auto-recovery

Field reports confirmed YouTube works on `youtube-discord-multidisorder`
but Discord-CF (`162.159.x.x:443`) stays dead on every variant — both
`tls-multisplit` and `tls-multidisorder` fail there because CF's edge
reassembles before classification, so TSPU sees the real SNI either way.

**Schema change**: `Profile::strategies.fake_multisplit` is now
`Vec<FakeMultisplitCfg>` instead of `Option<FakeMultisplitCfg>`. This
allows pushing one entry per Discord target with per-target tuning.
TOML format also changed: `[strategies.fake_multisplit]` (single
table) → `[[strategies.fake_multisplit]]` (array of tables). Old
TOML profiles with a single section now parse as zero entries due to
`#[serde(default)]` on the field — they were already obsolete (same
pre-2026-05 ALT11 single-target lock-in) so this is acceptable.

**`FakeMultisplit` strategy gating**: was hardcoded to
`Target::DiscordGateway`. Now honours a `target_filter: Vec<Target>`
field, falling back to the legacy hardcoded behaviour when empty.

**Default profile gets a CF-gentle fake_multisplit entry**:
`fooling=ttl=4 repeats=2 decoy_host=vk.me split_pos=1 seqovl=0
targets=[DiscordCloudflare]`. Mechanism: emit 2 fake ClientHellos
with SNI=vk.me (RU host TSPU whitelists, per Flowseal v4.5
08.01.2026) at TTL=4 so they die ~4 hops out and never reach CF
edge. TSPU sees the fake first, locks the per-flow classifier on
vk.me. Real CH then arrives untouched, CF accepts it, handshake
completes. Solves point #2 from the gap list below as a special
case (without full `fake-tls-mod` knob).

**`default_ru_decoy_host()` changed**: `max.ru` → `vk.me`. Flowseal
switched in v4.5 (08.01.2026) and `vk.me` has held since. Fallback
profiles can still pin to `max.ru` via explicit `decoy_host` in the
TOML if a region-specific TSPU misbehaves on vk.me.

This is **per-target auto-selection** — one engine, one profile,
the right strategy per classified flow. No profile switching, no
detection harness; the `tls_multisplit.targets` and
`fake_multisplit.targets` filters route at packet time. Adaptive
runtime profile rotation is still in scope for v0.4 (gap #4 of
"things the engine still cannot do" below) but is no longer the
critical path.

In `profiles/youtube-discord-2026-05.toml`, an external profile users
can run today via `sonicdpi run --profile …` with hostlists trimmed
to v4.6's set and Discord voice port set narrowed to `50000-50032`
(was `50000-50100` — Flowseal narrowed it in v3.9 / 12.08.2025 after
the upper half stopped being used by the actual Discord voice CDN).

## What the engine still cannot do (v0.3 work)

Each of these is a specific zapret feature that the current
`Strategy` trait + supporting types don't model. Adding any one of
them is a self-contained PR.

1. **`tls_multidisorder` for the real ClientHello.**
   Today `TlsMultisplit` emits split segments in order with a seqovl
   pattern. Multidisorder = same split, send segments out of order,
   re-tx the first segment last. zapret's classifier-confusion edge
   has shifted toward this since v4.5. New module
   `crates/sonicdpi-engine/src/tls_multidisorder.rs`; new TOML key
   `[strategies.tls_multidisorder]`; reuse the seqovl-pattern
   resolution from `embedded_or_synth_ch`.

2. **`fake-tls-mod` knob on fake decoys.**
   Right now the fake ClientHello emitted by `FakeMultidisorder` is
   either an embedded `.bin` (captured), `build_fake_clienthello`
   synthetic, or a user file. v4.6 mutates it on the fly with
   `rnd,dupsid,sni=X`: rnd = randomize TLS extension order, dupsid
   = duplicate the session_id field, `sni=vk.me` = rewrite the SNI
   extension to `vk.me`. The first two confuse signature-based
   pattern match; SNI=vk.me makes the fake hit a TSPU whitelist.
   New helper `apply_tls_mod(&mut Vec<u8>, &str)` in
   `crates/sonicdpi-engine/src/fakes.rs`. Apply on the path that
   builds the fake payload, not on the hot path.

3. **`badseq-increment` / `badack-increment`.**
   `Fooling::Badseq` is parameter-less. v4.6 uses
   `badseq-increment=0 / badack-increment=1` (i.e., wrong ack only,
   keep seq correct). Extend `Fooling::Badseq` to
   `Fooling::Badseq { seq_inc: i32, ack_inc: i32 }`; update
   `parse_fooling` to accept `badseq[+inc=N][,badack=N]`; update the
   per-fake mutation site in `fooling.rs::apply_fooling`.

4. **`fakeknown` / `fakedsplit` modes.**
   Used by v4.6 in the Games(TCP) lane. `fakeknown` = only emit fakes
   for the protocols we positively classified. `fakedsplit` = a
   weaker split that only wraps the first byte and copies it. Low
   priority for YouTube/Discord but listed here so the next reviewer
   doesn't have to re-research.

5. **`syndata` for Cloudflare WARP gateway.**
   Inject a payload during the SYN itself so DPI sees data before
   the connection is established. Gates on TCP-state == SYN-sent;
   `Strategy::on_packet` doesn't currently see SYNs because we
   filter them out in `flow.rs`. Adding this requires plumbing SYNs
   into the pipeline and a new `SynData` strategy.

6. **HTTP/80 `fake-http` path for googlevideo.**
   The engine currently only mangles TLS (port 443) and QUIC
   (UDP/443). v4.6 sends a fake TLS ClientHello on TCP/80 to
   confuse the HTTP classifier into seeing TLS — TSPU's L7
   classifier branches on bytes 0x16 0x03 0x0X. Add a new HTTP-80
   strategy that emits the same `tls_clienthello_www_google_com.bin`
   as a fake-with-TTL-fooling. New target type `YouTubeHttp` so
   we don't apply this to all of HTTP.

7. **Discord wide TCP port set.**
   `InterceptorConfig::tcp_ports` is a `Vec<u16>`. Default is
   `[80, 443]`. v4.6 needs `[80, 443, 2053, 2083, 2087, 2096, 8443]`.
   This is just a default-value change in `cli/main.rs`, but it
   ripples into the platform backends:
   - WinDivert filter string template (filter on port set).
   - Linux nftables rule installer (port range / set).
   - macOS pf rdr-to (port set).
   Each backend has a one-liner to update; do them together to
   keep the matrix consistent.

8. **DNS-over-HTTPS (port 853) lane.**
   Cloudflare WARP gateway listens on 853 too. Same packet path as
   443, just an extra port to intercept. Trivial once #7 is done.

## Verification checklist when v0.3 lands

- `cargo test --workspace --lib` still passes.
- `sonicdpi probe -H rr1.googlevideo.com --profiles youtube-discord` returns
  TLS handshake under 1.5s on a known-blocked ISP.
- `sonicdpi probe -H gateway.discord.gg --profiles youtube-discord` succeeds.
- Discord voice channel: actual round-trip RTC media (not just gateway
  reachability) — observed via `wireshark` `udp.port == 19294-19344`.
- `cargo deny check licenses` clean.
