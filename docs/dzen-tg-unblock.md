# Свой SOCKS5-прокси для Telegram на Rust — через официальный WebSocket-транспорт. 6 МБ, без серверов, opensource

> **TL;DR:** написал десктопное приложение `tglock` на Rust, которое поднимает локальный SOCKS5-прокси на 127.0.0.1:1080 и заворачивает Telegram-трафик в WebSocket-соединения с `web.telegram.org` — официальный WebSocket-транспорт MTProto, который описан в документации, но почти никем не используется в десктоп-клиентах. Получился стабильный канал с низкой задержкой без своего сервера и без подписки. Код целиком на GitHub, бинарник весит 6 МБ.

## С чего всё началось

Весной у меня Telegram Desktop стал жить странной жизнью: сообщения отправляются с задержкой по несколько секунд, медиа подгружается со второй-третьей попытки, голосовые звонки рвутся посреди разговора. На том же канале связи Discord, YouTube и условный SoundCloud — работают как обычно. То есть это что-то конкретно по подсетям Telegram (`149.154.x.x`, `91.108.x.x`).

Первая мысль: «ну поставлю VPN, делов-то». Но для одного мессенджера гонять весь трафик через сторонний сервер — оверкилл. Платный — это +200–300 ₽/мес и просадка скорости в браузере. Бесплатный — это вопрос, что они со мной делают и кому я плачу данными вместо денег. Хочется решения, которое:

- работает только для Telegram, не трогая остальной трафик
- не требует своего сервера
- не требует абонентки
- не дорабатывается напильником при каждом обновлении

И вот тут в дело пошла документация.

## Находка в документации MTProto

Я полез в [официальную документацию протокола Telegram](https://core.telegram.org/mtproto/transports#websocket) и наткнулся на раздел **WebSocket Transport**:

> Implementation of the WebSocket transport is pretty much the same as with TCP. Sec-WebSocket-Protocol must be set to "binary". All data received and sent through WebSocket messages is to be treated as a single duplex stream of bytes, just like with TCP.

То есть Telegram официально поддерживает WebSocket как альтернативный транспорт. И веб-версия `web.telegram.org` — это не просто браузерный клиент, это полноценные точки входа в сеть Telegram. Каждый Data Center имеет свой именованный WebSocket-эндпоинт:

| DC | Имя | URL |
|---|---|---|
| 1 | Pluto | `wss://pluto.web.telegram.org/apiws` |
| 2 | Venus | `wss://venus.web.telegram.org/apiws` |
| 3 | Aurora | `wss://aurora.web.telegram.org/apiws` |
| 4 | Vesta | `wss://vesta.web.telegram.org/apiws` |
| 5 | Flora | `wss://flora.web.telegram.org/apiws` |

Я прогнал прямые `ping`/`curl`-тесты:

- `149.154.167.51:443` (DC2 IP) — задержка 200+ мс, нестабильно.
- `web.telegram.org` (он же — Cloudflare фронт) — стабильные 50–80 мс по обычному HTTPS.

Разница объясняется просто: к адресам Telegram-подсетей мой провайдер относится «специально», а к Cloudflare — никак, потому что там же половина интернета. Если завернуть MTProto в WebSocket-фрейм поверх HTTPS-соединения с `*.web.telegram.org`, провайдер видит просто HTTPS-трафик к одному из Cloudflare-CDN-узлов. Это уже рабочая гипотеза.

## Архитектура: SOCKS5 → WebSocket

```
Telegram Desktop
       │
       ▼ SOCKS5
┌──────────────────┐
│   tglock         │  127.0.0.1:1080
│   локальный      │
│   прокси         │
└──────┬───────────┘
       │
       ├── IP Telegram-подсети?  ──►  WSS к {dc}.web.telegram.org/apiws
       │                              (на проводе: HTTPS к Cloudflare)
       │
       └── Любой другой IP?      ──►  Прямой TCP passthrough
```

Telegram Desktop умеет работать через SOCKS5-прокси (`Settings → Advanced → Connection type`). Я написал свой SOCKS5-сервер на 127.0.0.1:1080, который:

1. Принимает SOCKS5-handshake от Telegram Desktop.
2. Смотрит на IP назначения. Если IP попадает в Telegram-подсети — заворачиваем в WebSocket. Если нет (CDN, стикеры, апдейты) — прямой TCP без обработки.
3. Открывает WSS-соединение к нужному DC и связывает байты в обе стороны.

## Почему Rust

- **Один бинарник ~6 МБ**, без рантайма, без зависимостей. Положил на флешку — носи с собой.
- **Низкая задержка.** Прокси не должен быть бутылочным горлышком. Async I/O через `tokio` даёт latency ниже 1 мс на handshake.
- **Cross-compile.** Из коробки собирается под Linux, macOS, Windows, ARM64. Тот же код, разные таргеты.
- **Компилируется — значит работает.** Это не пустой слоган: после первого `cargo build --release` я ни разу не словил рантайм-крэш в проде.

Альтернативы я тоже взвешивал. Electron — 200 МБ бандл и тяжёлый запуск. Python — нужен интерпретатор и pyqt. Go — отличный вариант, но я хотел compile-time гарантии и `Result<T, E>`-стиль обработки ошибок, который у меня лучше укладывается в голове.

## Реализация

Приложение состоит из четырёх модулей:

| Модуль | Что делает |
|---|---|
| `main.rs` | GUI на `egui` + управление жизненным циклом прокси |
| `ws_proxy.rs` | SOCKS5-сервер + WebSocket-туннель |
| `bypass.rs` | DNS-настройка, системные утилиты |
| `network.rs` | Сетевая диагностика |

### SOCKS5-handshake

Когда Telegram Desktop коннектится через SOCKS5, происходит четыре шага:

```
// Клиент → сервер: [0x05, 0x01, 0x00]   — SOCKS5, 1 метод, no-auth
// Сервер → клиент: [0x05, 0x00]         — принято
// Клиент → сервер: [0x05, 0x01, 0x00, 0x01, IP, PORT]  — CONNECT
// Сервер → клиент: [0x05, 0x00, 0x00, 0x01, 0x00x4, PORTx2] — OK
```

Это всё в `tokio::net::TcpStream` укладывается в десяток строк.

### Определение DC по IP

В документации Telegram приведены подсети для каждого DC. Маппинг стабильный, апдейтится раз в квартал:

```rust
fn telegram_dc(ip: Ipv4Addr) -> Option<u8> {
    let o = ip.octets();
    match (o[0], o[1]) {
        (149, 154) => Some(match o[2] {
            160..=163 => 1,    // DC1
            164..=167 => 2,    // DC2
            168..=171 => 3,    // DC3
            172..=175 => 1,    // DC1 alt
            _         => 2,
        }),
        (91, 108)  => Some(match o[2] {
            56..=59  => 5,     // DC5
            8..=11   => 3,     // DC3
            12..=15  => 4,     // DC4
            _        => 2,
        }),
        (91, 105)  => Some(2),
        (185, 76)  => Some(2),
        _          => None,
    }
}
```

Вариант хитрее — парсить первые 64 байта `obfuscated2`-handshake и доставать DC ID оттуда. Это точнее, но это отдельный мини-проект. Пока хватает маппинга по IP.

### Открытие WSS-соединения

Обязательный заголовок `Sec-WebSocket-Protocol: binary` — без него сервер закрывает соединение сразу. Это прямо в первом абзаце доки, я на нём потерял пару часов когда копи-пастил пример из туториала по WebSocket в браузере.

```rust
let mut request = ws_url.as_str().into_client_request()?;
request.headers_mut().insert(
    "Sec-WebSocket-Protocol",
    "binary".parse()?,
);

let connector = tokio_tungstenite::Connector::NativeTls(
    native_tls::TlsConnector::new()?,
);
let (ws, _) = tokio_tungstenite::connect_async_tls_with_config(
    request, None, false, Some(connector),
).await?;
```

`native-tls` тянет системные сертификаты Windows — без привязки к OpenSSL и без необходимости таскать с собой `cacert.pem`.

### Двунаправленный relay

Главное наблюдение из доки: Telegram гарантирует, что данные в WebSocket — это «такой же поток байтов, как в TCP». Парсить MTProto-фреймы не нужно. Просто переливаем байты:

```rust
// TCP → WebSocket
let up = async {
    let mut buf = vec![0u8; 32768];
    loop {
        match tcp_rx.read(&mut buf).await {
            Ok(0) => break,
            Ok(n) => {
                let msg = Message::Binary(buf[..n].to_vec());
                if ws_tx.send(msg).await.is_err() { break; }
            }
            Err(_) => break,
        }
    }
};

// WebSocket → TCP
let down = async {
    while let Some(Ok(msg)) = ws_rx.next().await {
        if let Message::Binary(data) = msg {
            if tcp_tx.write_all(&data).await.is_err() { break; }
        }
    }
};

tokio::select! { _ = up => {}, _ = down => {} }
```

Здесь `tokio::select!` обрывает обе таски как только одна сторона закрылась — без этого можно было бы залипнуть с одной стороной open и второй closed.

## GUI на egui — без Electron

Нативный десктопный интерфейс через [`egui`](https://github.com/emilk/egui) / `eframe`. Никакого браузера, никакого JavaScript, immediate mode rendering на 60 FPS.

Окно простое:

- **Кнопка «Запустить»** — поднимает SOCKS5-сервер, переключает DNS на 1.1.1.1 для надёжности резолва `*.web.telegram.org`.
- **Кнопка «Настроить Telegram автоматически»** — открывает Telegram Desktop по схеме `tg://socks?server=127.0.0.1&port=1080`, клиент сам подхватывает прокси.
- **Лог-окно** — показывает диагностику в реальном времени: сколько туннелей открыто, к каким DC.

В сумме — около 700 строк UI-кода, включая обработку всех состояний, плюс ~600 строк сетевого кода. Не пугающе.

## Грабли, на которые я наступил

### Не-Telegram трафик через SOCKS5

Telegram Desktop, оказывается, через прокси кидает не только сам MTProto, но и запросы к стикер-серверам, обновлениям, превью-картинкам. Эти ребята живут на отдельных IP, никаких WebSocket-эндпоинтов для них нет. Если упрямо заворачивать всё подряд — половина клиента ломается. Решение: проверяем destination IP по маппингу, и для любого нерасспознанного просто прокидываем TCP без модификации.

### Graceful shutdown

Когда юзер жмёт «Стоп» — нужно: закрыть все открытые WebSocket-соединения, вернуть DNS на DHCP, не оставить Telegram Desktop без связи на полпути сообщения. Сделал через `Arc<AtomicBool>` флаг остановки, который проверяют все активные таски в своих `select!` ветках.

### TLS на Windows

Сначала пробовал `rustls` — компилируется, но не дружит с Cloudflare-сертификатом по необъяснимой причине. Замена на `native-tls` (использует системный SChannel) решила всё за пять минут. Когда работаешь с Windows — иногда лучше говорить системе её родным языком.
